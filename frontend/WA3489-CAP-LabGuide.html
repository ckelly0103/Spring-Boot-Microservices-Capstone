Project Overview

Module 1

For this project, you will use established Microservices techniques to design and implement REST services to support a customer management application.

1.1. Overview

The REST API Project focuses on:

  • Building a RESTful API using the Java Spring Boot framework
  • Crafting the API to support a customer management front-end application
  • The use of Java, Spring Boot and Gradle
  • Testing the API using Postman
  • Containerized Deployment of the API using Docker

1.2. High Level Requirements

The front-end application requires the REST API to support the following features:

  • Customer registration
  • Customer Login
  • Display, create, update and delete of customer records

1.3. Development and Testing Environment

Your development environment should include:

  • Git
  • Java
  • Gradle
  • A Java IDE (Eclipse)
  • Postman REST Client
  • Node.js (for the front-end application)
  • Web Browser (for testing the front-end application)
  • Docker (for containerized deployment of the API)

The above software is pre-installed on the Linux Project VM.

1.4. The Project Zip

The project zip file contains project related files that you’ll need to complete the project. It should be available in the student portal. If needed check with your instructor as to the zip file’s exact name and location.

The zip file contains a copy of a React based customer management front-end application.

\project-overview\wa3489-cust-front-end

The front-end application is provided to help you test your REST API. You will not be required to modify or complete the front-end application as part of this project.

Application Design

Module 2

This REST APIs you create as part of this project need to support the following functions for a customer management application:

  • Register new customers
  • Login existing customers
  • Create, Update and Delete Customer records

2.1. Component Diagram

2.2. Component List

  • Authentication Service
    • The client application uses this service to register new customers.
    • The client application uses this Service to obtain a JWT token
  • Data Service (customer management API)
    • The client application uses this service to:
      • Retrieve customer records
      • Create new customer records
      • Update existing customer records
      • Delete customer records
  • Client Application
    • The client is a React application that provides a user interface for the customer management application.

2.3. The Client Application

The client application is provided in the wa3489-front-end directory of the project zip file. A similar application was the focus of the previous capstone project.

When you’ve completed the services in this capstone you’ll run the provided client application using the REST APIs you created to verify their functionality.

2.4. Account Service

The Account Service is a new service that you will create to support the authentication of customers. It will provide a token endpoint that the client application can use to obtain a JWT token for accessing the Data Service.

The following information describes the interface that the Client Application expects to use when connecting to the Account Service.

  • Service information:
    • context path = \account
    • port = 8081
  • Endpoints
  • The Root Endpoint
    • Handles requests send to the \account\ path
    • Should return a message confirming that the service is up and running. This is useful for testing that the service is running.
  • The Token Endpoint
    • Responds to requests sent to the \account\token path
    • Is used to log in to the application.
    • Takes username and password as parameters
    • Checks that username exists in customer store
    • Checks that password matches the one listed for the customer in the customer store
    • Returns a JWT token suitable for connecting to the Data Service.
    • A token is not required to access the account service endpoints
  • The Register endpoint
    • Responds to requests sent to the \account\register path.
    • Takes name, email and password and adds them to the Customer data store via the Customer endpoint of the data service
  • Security.
    • A token is not required to access the account service endpoints

2.5. Data Service

  • Service information
    • context path = \api
    • port = 8080
  • Backend Data
    • Will use embedded hsql database
  • Endpoints
    • \api\customers
    • \api
  • Root Endpoint (\api )
    • Calling this endpoint returns a message confirming that the service is up and running.
  • Customers Endpoint (\api\customers )
    • Supports CRUD management of Customer records
    • Supports GET, PUT, POST and DELETE requests
    • A JWT token is required for all requests
  • Security
    • Use a servlet filter to intercept calls to the API and check requests for JWT tokens

2.6. Design Note

This application described has been simplified so that it’s achievable during the capstone week. In a production system you would typically see additional features such as:

  • Separate logins for customers and employees.
  • A separate service for managing employees who manage customer records.
  • Customers would be restricted to only being able to view and update their own records.
  • Employees would be able to view all customer records but only employees with 'admin'.access would be able to create, update or delete customer records.

These features are not included in this project to keep the scope manageable.

If time permits, and you’ve completed the features required for the capstone, you can look into implementing these features as well.

2.7. High Level Work Steps

For this capstone you are free to work in any order you like. However, the tasks in the order listed below is suggested as a guide.

  1. Create Customer CRUD endpoints

  2. Test the Customer endpoints

  3. Create the Account register endpoint

  4. Test the register endpoint

  5. Create the Account token endpoint

  6. Test the token endpoint

  7. Secure the Customer endpoints with a filter that checks for existence and content of JWT tokens

  8. Create a token and test Customer endpoints including token

  9. Test APIs using the React client application.

  10. Create Dockerfiles for the front and back end and use them to create working Docker images.

  11. Bring up the back-end API using Docker

  12. Verify that the API functions correctly when run using Docker

2.8. Deliverables

To prove your work you should be prepared to show:

  1. All the services working when tested with Postman.

  2. An unmodified version of the provided front end application working with your APIs.

  3. The APIs running in a Docker environment.

  4. The Account API returning a JWT token when the correct credentials are provided.

  5. The Customer API accepting or rejecting requests based on the 'Authorization' header.

This is a basic list of the deliverables. Your instructor may ask you to demonstrate additional features or functionality.

2.9. Application Q & A

Here are some frequently asked questions about the application design and implementation. Before asking your instructor a question, think through your question first. You may find that you can answer it yourself by reviewing the design and implementation steps.

Q01 - Should I be using Git during the project?

A01 - Yes, you should be using Git to manage your code. You can create a local Git repository for each of the services as soon as you start working on them. This will allow you to track changes and revert to previous versions if necessary. Make sure you commit your changes regularly, especially after completing significant tasks or features.

Q02 - Do I need to push my changes up to a remote repository like GitHub?

A02 - It depends on your course requirements. If your instructor has asked you to push your changes to a remote repository, then you should do so. Otherwise, you can keep your changes in a local repository until the end of the project.

Q03 - Should account and the customer endpoints be the same or separate Spring Boot applications?

A03 - It’s up to you. You can implement them as separate Spring Boot applications or as a single application. Before you start working you should think through the advantages and disadvantages of each design. If you choose to implement them as separate applications, make sure they can communicate with each other using the appropriate protocols (e.g., HTTP).

Q04 - How can I test the APIs?

A04 - Get requests are easily tested in a browser. For POST, PUT and DELETE requests you can use Postman. The React client application can also be used to test the APIs.

Q05 - It sounds like the provided client application can only be used once all the APIs have been implemented. Is that correct?

A05 - Yes and No. As implemented, the client application only works once the APIs have all been implemented. But the client is just a React application. If you want to you could easily modify it to check the various endpoints as you are working on them.

Extending the Application

Module 3

If you have time after implementing the capstone’s initial requirements, you can consider extending the application. This is also something you could do after the course is completed to help you practice the programming skills you’ve learned.

This section outlines a potential extension to the application: an event management system. Alternatively, as previously mentioned, the application could be updated to support employee users. Or, if you have other ideas, you can implement those as well.

Reading the section below can be of help even if you decide to go with one of your own ideas. Either way you’ll want to think through the requirements, design and work plan before you start coding.

3.1. Implementing an Event Management System

This section describes the design and work plan needed to extend the original application to include Event management services. Such a system could be used to allow conference attendees to sign up for presentations, workshops, or other events. Even if you don’t implement this extension, the design and work plan can be used as a guide for implementing other features in the future.

Any changes at this point are likely to require changes to the front and back-end applications. The front-end application will need to be updated to allow users to view and register for events, while the back-end will need to provide APIs to manage events and registrations.

The system requires additional REST APIs that:

  • Create, Update and Delete Event records
  • Create, Update and Delete Registration records.

Possible User Workflow:

  • User registers themselves in the client application
  • User then Logs into the client application
  • User views a list of events they have registered for.
  • A button allows the user to view a list of all available events
  • To register for an event the user selects and event and clicks on a 'Register' button.
  • The user is presented with a form to enter their registration details and clicks on a 'Complete Registration' button.
  • User creates a registration record for events they want to attend.
  • Events themselves can only be added and deleted by an Admin user.

The Registration API is not the same thing as the Register API that you created earlier. The Registration API is used to manage the relationship between users and events, while the Register API is used to create new user accounts.

3.3. New Components

  • Registrations API Service

3.4. Component Interaction

  • The client application gets event records from the Event service and displays them
  • The client application uses the Registration service to create records linking users to Events
  • The client application can display a list of all the events a user has registered for.
  • The client application allows users to modify or delete previously created Registrations
  • The client application allows users to create, update and delete Events (if they log in as an Admin user)

  • Events Endpoint

3.6. Suggested Work Plan

When coding you should always know what you are trying to accomplish. It’s more effective to work in small steps, completing one task at a time, than to try and do everything at once. Although they are not explicitly mentioned here you should always be thinking about how you will test your code as you go along.

  1. Create schema and initial data for Event API

  2. Create Event API and Code CRUD endpoints

  3. Create schema and initial data for Registrations API

  4. Create Registrations API and Code CRUD endpoints

  5. Add an Event List page to the client application

  6. Add a registrations list page to the client application

  7. Add a form for adding/modifying/deleting Events

  8. Add a form for adding/modifying/deleting Registrations

  9. Test the new Event and Registration features
